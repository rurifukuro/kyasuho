  - iOS: `app.json` の infoPlist に `NSFaceIDUsageDescription` 必須（ないと審査リジェクト）。文言は日本語+英語。
  - 対象はオーナー・キャストの**両方のログイン系統**（T13の2系統どちらでも使える）。

- **キャスト個人がアプリ側で「年間の給与」「各月の給与」を把握できる**ようにする＝管理者側の給与計算（§3-F・migration 0009 `ky_payroll` 系）と**同じデータに連携**する（別計算を作らない＝数字がズレたら事故）。
- T13の基盤前提。RLS: キャストは**自分の給与行のみ select**（店全体・他キャストは不可）。
- キャストUIに「給与」画面: 月別一覧（当月〜過去12ヶ月）＋年間合計。表示項目・計算式は管理側（SPEC §23）と完全一致させる。管理側で未確定の月は「集計中」等の状態表示。

#### T16. 店舗独自テンプレートの取り込み（月間スケジュール＋日次出勤表・2026-07-06 ユーザー指示）

- **各店舗が既に持っている自前のテンプレートデザインを取り込んで**、月間スケジュール表（§3-I シフト表エンジン）と日次出勤表（T8）の**両方**の生成に使えるようにする。内蔵テンプレ20種・AIデザインに並ぶ第3の選択肢「店舗テンプレート」。
- 想定フロー: 店側が管理画面（アプリ/管理Web）でテンプレ画像をアップロード→**背景・飾り枠として使い、その上に出勤キャスト（写真/名前/時間）を重ねる配置領域を指定**（グリッド範囲・行列数・タイトル位置などの簡易エディタ。まずは矩形指定＋行列数入力の最小UIでよい）→保存すると以後テンプレ一覧に「うちの店のテンプレ」として並ぶ。
  - 発展形（余力があれば）: 既存のAIシフトデザイン（`ky-shift-design`）を拡張し、アップロード画像から配置領域をAIが推定して初期値を自動セット。
- 保存先: Supabase Storage（tenant毎パス＋RLS。T9のキャスト写真バケットと設計を揃える）。
- **取り込みテストはユーザー提供の見本2枚を必ず使うこと（2026-07-06 ユーザー指示）**:
  - 月間: `docs/shop_template_reference/monthly_schedule_sample.jpg`（Belle Étoile・白基調パール/ダマスク装飾・上部に店ロゴ＋営業時間・SUNDAY〜SATURDAYの7列グリッド・各日セルにキャスト名+時間の縦積み）
  - 日次: `docs/shop_template_reference/daily_lineup_sample.jpg`（T8見本と同一・黒ダマスク＋金額縁・3×3写真グリッド）
  - **この2枚は実在店舗・実在人物の写真＝gitignore済み・コミット/公開/SNS出力の見本として外に出すこと厳禁**（取り込み動作のローカル検証専用）。
- T8のテンプレ選択制・出勤人数に応じた可変グリッドの要件はT16の店舗テンプレにもそのまま適用（人数がグリッド数を超える/足りない日でも崩れない）。

#### T17. キャスト人物像・店舗遍歴管理（オーナー向け・移籍時の引き継ぎ資料）★T13前提

- **オーナーが各キャストに対して人物像メモ・評価を記入できる**（性格・強み・接客の特徴・お客様からの評判・育成メモ等のフリーテキスト＋構造化項目）。
- **店舗遍歴（ワークヒストリー）**: キャストがこれまでどの店で働いてきたかの記録。**他店に移籍した時に新しいオーナーが参考にできる**ようにする（2026-07-06 ユーザー指示）。
- 実装方針:
  - `ky_cast_evaluations`（テナント×キャスト・オーナーが記入）: persona_notes（人物像フリーテキスト）/ strengths / areas_for_improvement / customer_feedback_summary / internal_notes（社外秘メモ＝移籍先に**見せない**）。
  - `ky_cast_work_history`（キャスト単位・テナント横断）: cast_user_id / tenant_name / position / start_date / end_date / notes / visibility（`public`=移籍先にも見える / `private`=本人と記入者のみ）。
  - 遍歴はキャスト退店時（user_id解除時）に自動で1行追加する運用、または手動登録の両対応。
  - **RLS設計（重要・T13の延長）**:
    - オーナー: 自テナントのキャストの evaluations を CRUD。work_history は visibility=public の行を read（新キャスト加入時の参考用）。
    - キャスト本人: 自分の work_history を read（全行）。evaluations は internal_notes 以外を read（自己評価の参考）。
    - **internal_notes は記入テナントのオーナーのみ read（移籍先・キャスト本人には非公開）**。
  - 管理側UI: CastsScreen の各キャスト詳細に「人物像」タブ/セクション追加（フリーテキスト＋項目別入力フォーム）。遍歴は一覧表示（キャスト加入時に新オーナーが自動で閲覧可能）。

#### T18. キャスト個人情報入力（面接書類代替・キャスト本人入力）★T13前提

- **キャスト個人が自分の個人情報を入力する欄を設け、面接の書類をあまり記載しなくていいようにする**（2026-07-06 ユーザー指示）。
- 標準的な面接書類（履歴書）の項目を参考に以下を実装:
  - **基本情報**: 本名（氏名）/ ふりがな / 生年月日 / 性別 / 住所 / 電話番号 / メールアドレス（Auth連携で自動取得可）
  - **緊急連絡先**: 名前 / 電話番号 / 続柄
  - **通勤**: 最寄り駅 / 通勤方法 / 通勤時間（分）
  - **給与振込口座**: 銀行名 / 支店名 / 口座種別（普通/当座）/ 口座番号 / 名義（カナ）
  - **勤務希望**: 希望勤務日数（週） / 希望時間帯 / 勤務開始可能日
  - **経歴**: 過去の勤務経験（店舗名・期間・ポジション）← T17のwork_historyと連携可
  - **資格・免許**: フリーテキスト
  - **特記事項・備考**: フリーテキスト（アレルギー・持病・配慮事項等）
  - **顔写真**: T9のphoto_urlと共用
- 実装方針:
  - `ky_cast_personal_info`（cast_user_id単位・テナント非依存＝移籍しても引き継ぐ）: 上記項目のカラム。**PII（個人情報）のため取り扱い要注意**。
  - **RLS**: キャスト本人のみ write（自分の行の CRUD）。**現在所属テナントのオーナーのみ read**（`ky_casts.user_id` 経由でテナント判定）。退店後は read 不可（個人情報保護）。
  - **⚠️ PII保護**: 給与振込口座・住所・電話番号等は**暗号化保存を検討**（Supabase Vault または pgcrypto）。最低限、RLSで関係者以外のアクセスを厳密に遮断。App Privacy申告にも影響（収集データ項目の追加が必要＝R4）。
  - キャスト専用UI（T13のキャストホーム画面）に「個人情報」セクション追加。入力は段階的でOK（必須=本名/電話/緊急連絡先、任意=口座/資格等）。
  - オーナー側UI: キャスト詳細画面に「個人情報」タブ（read-only表示。口座番号等はマスク表示 `***1234`）。
